# Zibll OAuth2 接入开发文档

本文档定义了 Zibll OAuth2 代理服务的接口规范与安全建议。

## 1. 基础信息

- **Base URL**: `https://your-site.com/wp-json/zibll-oauth/v1`
- **认证模式**: 标准 OAuth2 Authorization Code Grant
- **安全传输**: 必须使用 `HTTPS`

## 2. 术语与参数

| 字段 | 说明 |
| :--- | :--- |
| `client_id` | 应用唯一标识（在 Provider 后台创建应用获取） |
| `client_secret` | 应用密钥（请勿泄露，仅限服务端使用） |
| `redirect_uri` | 授权回调地址（必须与后台配置完全一致，精确匹配） |
| `code` | 临时授权码（有效期 10 分钟，一次性消费） |
| `access_token` | 资源访问令牌（默认有效期 2 小时） |
| `refresh_token` | 刷新令牌（有效期 180 天，滑动过期，每次刷新会轮换） |

## 3. 核心流程

### 3.1 授权 (Authorize)
引导用户在浏览器访问：
`GET /authorize`

**请求参数:**
- `response_type`: 固定为 `code`
- `client_id`: 应用标识
- `redirect_uri`: 回调地址
- `state`: 接入方生成的随机串（用于防 CSRF，回调时原样返回）
- `scope`: 权限范围（默认 `basic`，可选 `email`, `profile`, `phone`）

**用户同意后回调:**
`{redirect_uri}?code={CODE}&state={STATE}`

> 财务授权（`/authorize_finance`）需要用户已完成一次应用授权（`/authorize`）。
> 若用户从未授权该应用，直接访问 `/authorize_finance` 将被拒绝。

### 3.2 换取令牌 (Token)
接入方后端通过 code 换取令牌：
`POST /token`

**请求参数 (Form 或 JSON):**
- `grant_type`: 固定为 `authorization_code`
- `client_id`: 应用标识
- `client_secret`: 应用密钥
- `code`: 回调获取的 code
- `redirect_uri`: 必须与授权时一致

**成功响应 (200 OK):**
```json
{
  "access_token": "AT_xxx",
  "token_type": "bearer",
  "expires_in": 7200,
  "refresh_token": "RT_xxx",
  "refresh_token_expires_in": 15552000
}
```

**错误响应:**
```json
{
  "code": "invalid_grant",
  "message": "code 无效或已过期",
  "data": {}
}
```

### 3.3 刷新令牌 (Refresh)
当 access_token 快过期时，使用 refresh_token 获取新令牌：
`POST /token`

**请求参数:**
- `grant_type`: 固定为 `refresh_token`
- `client_id`: 应用标识
- `client_secret`: 应用密钥
- `refresh_token`: 当前的 refresh_token

**说明:** refresh_token 采用**旋转机制**，每次刷新都会返回新的 `refresh_token`，旧令牌立即失效。

### 3.4 吊销令牌 (Revoke)
主动失效令牌：
`POST /revoke`

**请求参数:**
- `client_id`: 应用标识
- `client_secret`: 应用密钥
- `token`: 要吊销的令牌
- `token_type_hint`: 可选 `access_token` 或 `refresh_token`

**成功响应 (200 OK):**
```json
{
  "revoked": true
}
```

**错误响应:**
```json
{
  "code": "invalid_client",
  "message": "client_secret 无效",
  "data": {}
}
```

## 4. 用户信息接口

### 4.1 获取用户信息 (UserInfo)
`GET /userinfo`
Header: `Authorization: Bearer <access_token>`

**成功响应 (200 OK):**
```json
{
  "userinfo": {
    "openid": "o_xxx",
    "name": "用户名",
    "avatar": "https://...",
    "email": "user@example.com",
    "phone": "+8613800000000",
    "description": "个人简介",
    "balance": 100.50,
    "refresh_token": "rt_xxx",
    "refresh_token_expires_in": 15552000
  }
}
```

### 4.2 获取 UnionID (UnionID)
`GET /unionid`
Header: `Authorization: Bearer <access_token>`

**成功响应 (200 OK):**
```json
{
  "openid": "o_xxx",
  "unionid": "123"
}
```

## 5. 财务扣款 (异步模型)

本服务提供免密扣款能力，扣款过程为异步执行。

### 5.1 发起扣款 (Deduct)
`POST /finance/deduct`
Header: `Authorization: Bearer <access_token>`

**Body 参数:**
- `product_name`: 商品名称 (必填)
- `amount`: 金额 (必填，必须 > 0)
- `order_no`: 你的订单号 (必填，幂等键，全局唯一)

**响应状态码:**
- `202 Accepted`: 订单已受理，正在异步处理
- `200 OK`: 订单已处理完毕（通常见于幂等重试时）

**响应示例 (202 Accepted - 处理中):**
```json
{
  "trade_no": "zod_xxx",
  "order_no": "ORD123",
  "status": "processing",
  "message": "订单已受理，处理中",
  "amount": 10.00
}
```

**响应示例 (200 OK - 成功):**
```json
{
  "trade_no": "zod_xxx",
  "order_no": "ORD123",
  "status": "success",
  "message": "订单已处理完成",
  "amount": 10.00,
  "balance": 90.50
}
```

**响应示例 (200 OK - 失败):**
```json
{
  "trade_no": "zod_xxx",
  "order_no": "ORD123",
  "status": "failed",
  "message": "订单处理失败",
  "amount": 10.00
}
```

### 5.2 查询结果 (Verify)
`GET /finance/verify`
Header: `Authorization: Bearer <access_token>`

**参数:** `trade_no` 或 `order_no`（至少传一个）

**成功响应 (200 OK):**
```json
{
  "trade_no": "zod_xxx",
  "order_no": "ORD123",
  "product_name": "商品名称",
  "amount": 10.00,
  "status": "success",
  "error_msg": "",
  "created_at": "2024-01-01 10:00:00",
  "updated_at": "2024-01-01 10:00:05"
}
```

**status 字段说明:**
- `processing`: 处理中
- `success`: 成功
- `failed`: 失败
- `unknown`: 未知

### 5.3 获取签约状态 (Sign Status)
`GET /finance/sign_status`
Header: `Authorization: Bearer <access_token>`

**成功响应 (200 OK):**
```json
{
  "openid": "o_xxx",
  "is_signed": true,
  "scope": "basic email",
  "finance_scope": 1,
  "status": 1,
  "created_at": "2024-01-01 10:00:00",
  "authorized_at": "2024-01-02 15:30:00"
}
```

**字段说明:**
- `openid`: 用户 OpenID
- `is_signed`: 是否已完成财务签约（status=1 且 finance_scope=1）
- `scope`: 用户授权范围
- `finance_scope`: 财务授权状态（1=已签约，0=未签约）
- `status`: 授权状态（1=有效，0=已撤销）
- `created_at`: 首次授权时间
- `authorized_at`: 财务授权时间

## 6. 服务端回调与签名安全

当异步任务（如扣款、撤销授权）完成时，Provider 会 `POST application/json` 到接入方。

### 6.1 扣款结果回调

当扣款完成（成功或失败）时，Provider 会向应用配置的回调地址发送通知。

**请求方法:** `POST`
**Content-Type:** `application/json`

**请求体示例:**
```json
{
  "appid": "app_xxx",
  "openid": "o_xxx",
  "order_no": "ORD123",
  "trade_no": "zod_xxx",
  "amount": 10.00,
  "status": "success",
  "balance": 90.50,
  "timestamp": 1704067200,
  "nonce": "random_str",
  "event_id": "evt_xxx",
  "sign2": "sha256_signature"
}
```

**字段说明:**
- `appid`: 应用标识
- `openid`: 用户 OpenID
- `order_no`: 外部订单号
- `trade_no`: 平台交易号
- `amount`: 扣款金额
- `status`: 扣款状态（`success`/`failed`）
- `balance`: 扣款后余额
- `timestamp`: Unix 时间戳
- `nonce`: 随机字符串
- `event_id`: 事件 ID（用于去重）
- `sign2`: 签名

**接入方应返回:**
```json
{
  "code": 0,
  "message": "success"
}
```

### 6.2 撤销授权回调

当用户撤销授权时，Provider 会向应用配置的回调地址发送通知。

**请求方法:** `POST`
**Content-Type:** `application/json`

**请求体示例:**
```json
{
  "appid": "app_xxx",
  "openid": "o_xxx",
  "user_id": 123,
  "finance_granted": 1,
  "status": "revoked",
  "revoked_at": 1704067200,
  "timestamp": 1704067200,
  "nonce": "random_str",
  "event_id": "evt_xxx",
  "sign2": "sha256_signature"
}
```

**字段说明:**
- `appid`: 应用标识
- `openid`: 用户 OpenID
- `user_id`: 用户 ID
- `finance_granted`: 财务授权状态（1=已授权，0=未授权）
- `status`: 状态（`revoked`）
- `revoked_at`: 撤销时间（Unix 时间戳）
- `timestamp`: Unix 时间戳
- `nonce`: 随机字符串
- `event_id`: 事件 ID（用于去重）
- `sign2`: 签名

**接入方应返回:**
```json
{
  "code": 0,
  "message": "success"
}
```

### 6.3 签名算法 (Sign2)
全链路仅支持 **V2 全字段签名**。算法如下：
1. 排除 `sign` 和 `sign2` 参数。
2. 将剩余所有参数按 Key 的 **ASCII 码升序排序**。
3. 将排序后的参数拼成 `key1=value1&key2=value2`（value 为数组或对象时转 JSON，字符进行 rawurlencode）。
4. 使用 `client_secret` 作为 Key，对该字符串执行 `HMAC-SHA256`。

### 6.4 幂等建议
回调 Payload 包含 `event_id`。接入方应：
1. **验签**: 验证 `sign2`。
2. **去重**: 检查 `event_id` 是否已处理过。
3. **处理**: 执行业务逻辑并返回 `200 OK`。

## 7. 错误码规范

| 错误码 | HTTP | 说明 |
| :--- | :--- | :--- |
| `invalid_request` | 400 | 参数缺失或格式错误 |
| `invalid_client` | 401 | client_secret 校验失败 |
| `invalid_grant` | 400 | code/refresh_token 无效或已过期 |
| `unauthorized_client` | 403 | 应用不存在或未启用 |
| `ip_not_allowed` | 403 | 调用 IP 不在白名单 |
| `deduct_failed` | 400 | 扣款失败（余额不足等） |
